# Contributing to Nawa Learn

First off, thank you for considering contributing to Nawa Learn! It's people like you that make Nawa Learn such a great platform for the community.

## Code of Conduct

By participating in this project, you are expected to uphold our [Code of Conduct](CODE_OF_CONDUCT.md). Please report unacceptable behavior to the project maintainers.

## How Can I Contribute?

### Reporting Bugs

Before creating bug reports, please check the existing issues as you might find out that you don't need to create one. When you are creating a bug report, please include as many details as possible:
- A clear and descriptive title.
- Steps to reproduce the issue.
- Expected behavior vs. actual behavior.
- Screenshots or console logs if applicable.

### Suggesting Enhancements

Enhancement suggestions are highly welcomed! When suggesting an enhancement:
- Explain how it would work.
- Keep the scope as narrow as possible, to make it easier to implement.
- Explain why this enhancement would be useful to most users.

### Code Contributions

1. **Fork the repository** and clone it locally.
2. **Create a branch** for your edits (`git checkout -b feature/amazing-feature` or `git checkout -b fix/issue-123`).
3. **Make your changes**. Ensure your code adheres to our styling guidelines (Next.js & Tailwind CSS best practices).
4. **Test your changes**. Make sure your changes do not break existing functionality.
5. **Commit your changes** using clear and descriptive commit messages.
6. **Push to the branch** (`git push origin feature/amazing-feature`).
7. **Open a Pull Request** against the `main` branch.

## Development Guidelines

- **Architecture**: We use Next.js App Router. Please follow the existing folder structure.
- **Styling**: We strictly use Tailwind CSS. Please avoid custom CSS unless absolutely necessary. Keep the UI rich, modern, and aligned with our existing aesthetics.
- **Component Design**: Build modular, reusable React components. Place them in the `src/components` directory.
- **Types**: We use TypeScript. Ensure your variables, props, and functions are properly typed. Avoid using `any`.

## Pull Request Process

1. Ensure any install or build dependencies are removed before the end of the layer when doing a build.
2. Update the README.md with details of changes to the interface, if applicable.
3. Your PR will be reviewed by at least one core team member. We may ask for modifications before merging.
4. Once approved, the PR will be merged into the main branch.

Thank you for contributing!
